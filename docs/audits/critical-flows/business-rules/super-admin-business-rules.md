## 1. Tenant Creation Rules

**Trigger**: `SuperAdminNovoLab.tsx` → `super-admin-create-tenant`

| Rule | Evidence | Notes |
|---|---|---|
| `nome` is mandatory | create-tenant/index.ts: `if (!nome) return errorResponse(400, ...)` | Only required field |
| `adminEmail` mandatory, must contain `@` | create-tenant/index.ts | |
| `adminNome` mandatory | create-tenant/index.ts | |
| `adminSenha` if provided must be ≥ 6 chars | create-tenant/index.ts | If omitted → email invite sent instead |
| `slug` auto-generated from `nome` if blank | create-tenant/index.ts: `slugify(slugInput \|\| nome)` | |
| `labCode` custom: 3-12 chars A-Z/0-9 | create-tenant/index.ts: `/^[A-Z0-9]{3,12}$/` | Auto-generated by trigger if omitted |
| `status` defaults to `"ativo"` | create-tenant/index.ts: `status: "ativo"` | Tenant is immediately live |
| `plano` defaults to `"free"` | NovoLab form: `EMPTY = { plano: "free" }` + create-tenant body | |
| A **default unit** is seeded | create-tenant: `unidades` INSERT with `tipo:"SEDE", padrao:true` | Failure is non-fatal (warn only) |
| **Default mapas de trabalho** seeded | `rpc("seed_default_mapas_for_tenant")` | Failure is non-fatal |
| **Default formas de pagamento** seeded | `rpc("seed_default_formas_pagamento_for_tenant")` | Failure is non-fatal |
| Admin user gets `role="admin"` | `user_roles` UPSERT after user creation | |
| `lab_code` is **immutable** after creation | create-tenant comment: *"lab_code é imutável APÓS este ponto"* | DB trigger enforces |

**⚠ Production impact**: Creates live Auth user + tenant immediately. No staging or approval step. If admin creation fails, tenant exists but has no users — orphaned state is possible.

## 2. Tenant Edit Rules

**Trigger**: `SuperAdminTenantDetalhe.tsx` → `super-admin-update-tenant`

| Field | Editable | Rule |
|---|---|---|
| `nome` | ✅ | Free text, no uniqueness check on update |
| `cnpj` | ✅ | Mask applied client-side; raw digits sent to server |
| `email_contato` | ✅ | No validation in update-tenant function |
| `telefone` | ✅ | Raw digits sent |
| `cidade`, `estado` | ✅ | Free selection |
| `slug` | ❌ | Displayed as read-only in detalhe (line 489) |
| `lab_code` | ❌ | Displayed as read-only (line 490); DB trigger prevents change |
| `status` | ✅ | Only `"ativo"` ↔ `"suspenso"` toggle in UI (line 262) |
| `plano` | ✅ (legacy field in `tenants.plano`) | Passed in same update-tenant body |
| `database_strategy` | ❌ | Managed via separate TenantDatabaseConfig component |

**⚠ Production impact**: Status change to `"suspenso"` immediately denies login for all users in that tenant (tenant_resolve returns `tenant_inactive`). There is **no grace period** and **no notification** to the tenant admin.

## 3. Tenant Suspension / Reactivation Rules

| Rule | Evidence |
|---|---|
| Toggle between `ativo` ↔ `suspenso` only | `handleToggleStatus` line 262: `const novo = tenant.status === "suspenso" ? "ativo" : "suspenso"` |
| `bloqueado` status exists in DB but NOT in UI toggle | StatusFilter allows "inativo" but toggle only cycles ativo/suspenso |
| Confirmation dialog shown before suspend | `suspendOpen` Dialog — but no typed confirmation, just a click |
| No audit log of who suspended a tenant | Neither `super-admin-update-tenant` nor DB trigger creates an admin audit row |
| Tenant resolve rejects suspended tenants | tenant-resolve: `if (t.status !== "ativo") return { ok: false, error: "tenant_inactive" }` |

**Gap**: `"inativo"` and `"bloqueado"` statuses exist (referenced in filter) but no UI path sets them — only reachable via direct DB update.

## 4. Tenant Deletion Rules

**Trigger**: `SuperAdminTenantDetalhe.tsx` (Danger Zone) → `super-admin-delete-tenant`

| Rule | Evidence |
|---|---|
| Must type exact tenant `nome` to confirm | delete-tenant: `if ((tenant.nome).trim() !== confirmName) return 400` |
| All non-super-admin Auth users deleted | loop over `profiles` → `auth.admin.deleteUser` |
| Data cascade via FK | `tenants` DELETE cascades to all child tables |
| Super-admin users are skipped (not deleted from Auth) | per-user `rpc("is_super_admin")` check |
| **No soft delete** | Hard delete, no recycle bin |
| **No pre-delete backup trigger** | Operator must manually run backup before deleting |
| **Synchronous operation** | Large tenants may timeout edge function |
| Navigate to /laboratorios on success | line 309 |

**⚠ Production impact**: IRREVERSIBLE. No automatic backup. Orphaned storage files (S3/Supabase Storage) are NOT cleaned up by this function.

## 6. Backup Rules

**Trigger**: `SuperAdminTenantDetalhe.tsx` tab "Banco de dados" → `super-admin-tenant-backup`

| Rule | Evidence |
|---|---|
| Three formats: `sql` (INSERT dump gzipped), `json`, `xlsx` | backup/index.ts |
| **Only available for `shared_db` tenants** | Returns 409 for `isolated_db`: *"backup é gerenciado pelo provedor externo"* |
| ~40 tables exported | `TENANT_TABLES` constant in backup function |
| SQL uses `SET session_replication_role = replica` | Allows bypass of triggers on restore |
| `ON CONFLICT DO NOTHING` for all INSERTs | Idempotent restore but may silently skip conflicts |
| Backup is streamed as blob to browser | No server-side storage of backups |
| No scheduled/automated backup | Manual only |

## 7. Admin User Management Rules

### Password Reset
| Rule | Evidence |
|---|---|
| Min 6, max 72 chars | reset-tenant-password: `checkPasswordAcceptable` |
| Target user must belong to tenantId | `profile.tenant_id !== tenantId → 403` |
| Target cannot be super_admin | anti-escalation check via `rpc("is_super_admin")` |
| HIBP (Have I Been Pwned) may block weak passwords | Handled with friendly error message |
| Optional email change in same call | `newEmail` field — syncs `profiles.email` |

### Impersonation
| Rule | Evidence |
|---|---|
| Generates one-time magic link | `auth.admin.generateLink({ type: "magiclink" })` |
| Targets the oldest `perfil="admin"` non-super-admin | eligible sort logic in impersonate function |
| Link opened in `window.open("_blank")` | handleImpersonate line 296 |
| **No expiry shortening** | Supabase default magic link TTL applies (1 hour) |
| **No audit trail** in DB | Only edge function log (structured JSON) |

### Edit Admin
| Rule | Evidence |
|---|---|
| Updates `nome`, `email`, `telefone` | `super-admin-update-tenant-admin` |
| Does NOT reset password | separate dialog |
| Admin identified by `snap.admin_user.user_id` from snapshot | |

## 8. Subscription / Billing Rules

| Rule | Evidence |
|---|---|
| Plans stored in `subscription_plans` table | super-admin-plans function |
| Only one plan can be `is_default = true` | plans function clears others before setting default |
| Plan `code` is immutable after creation | Dialog disables code field when `editing.id` exists (line 184 Planos.tsx) |
| Plan deletion blocked if tenants are using it | delete action checks usage (seen in plans function) |
| MRR is stored as `mrr_cents` in `tenant_subscriptions_billing` | Calculated/set at plan assignment |
| No Stripe/payment gateway integration | Internal tracking only |
| Trial period tracked via `trial_ends_at` | Display only — no enforcement found |
| `canceled_at` displayed as warning | No automatic suspension on cancellation |

## 10. Global Configuration Rules

| Rule | Evidence |
|---|---|
| Settings stored in `saas_settings` table as JSON blobs | Configuracoes.tsx `loadSetting`/`saveSetting` — direct client queries |
| Keys: `smtp_config`, `ai_config`, `s3_config`, `whatsapp_config` | |
| Secrets (passwords, API keys) stored in plaintext JSON in DB | UI claims "AES-256" badge (line 312) but no encryption observed in read/write code |
| AI config save merges with remote to avoid overwriting other provider's key | `saveAiProvider` fetches remote before merge (line 221) |
| SMTP, S3, WhatsApp changes take effect immediately | No validation before save (except SMTP host/user/fromEmail required) |
