# Risk Analysis — ResultadoDetalhe Flow

**Audit date:** 2025-07  
**Focus:** Clinical Safety, Data Integrity, Maintenance, Scalability

---

## 1. Clinical & Data Integrity Risks

### 1.1 Critical Value Bypass in Bulk Release
**Evidence:** `ResultadoDetalhe.tsx:629–677` (`handleLiberarTodos`)
The bulk release function iterates through exams and calls `updateAtendimentoExame` with `status: "finalizado"` directly. It **never** calls `getParametrosCriticosDoExame` or triggers the `CriticoConfirm` dialog. 
- **Risk:** Clinical results outside safe limits (e.g., potassium 7.0) can be released to the patient without medical notification or analyst justification.

### 1.2 Release of Unsaved / Stale Data
**Evidence:** `ResultadoDetalhe.tsx:567, 592`
The "Liberar" flow (`handleAnalisarLiberar` -> `executarLiberacao`) does not invoke or require a prior `handleSalvar`.
- **Risk:** If a user types a result but clicks "Liberar" without clicking "Salvar" first, the system persists the *existing* DB state (potentially empty or old) as "finalizado". The local UI state is not synchronized to the DB during the release call.

### 1.3 Signature URL Expiration
**Evidence:** `ResultadoDetalhe.tsx:118-137`
The S3 presigned URL for the analyst's signature is fetched once on component mount with a 1-hour TTL. 
- **Risk:** In long-running sessions (common in lab bancada), the URL will expire. Subsequent printing or PDF generation will result in broken images or missing signatures on clinical documents.

### 1.4 Hardcoded Auditor Attribution
**Evidence:** `ResultadoDetalhe.tsx:116, 262, 314, 397`
The analyst name defaults to `"Felipe Andrade Melo"` and is hardcoded into the local audit log entries.
- **Risk:** Regulatory non-compliance (RDC 786/2023). Actions performed by one professional are attributed to another in the UI-visible audit trail, undermining accountability.

## 2. Maintenance & Structural Risks

### 2.1 Monolithic God Component
**Evidence:** `ResultadoDetalhe.tsx` (2,619 lines)
The file manages: routing, DB state, local UI state, business logic (release/save), 10+ modals, and complex HTML/PDF template generation.
- **Risk:** High cognitive load for developers; high probability of side effects during refactoring; difficult to unit test.

### 2.2 Logic Duplication (Fragile Updates)
**Evidence:** `Complexity Audit Section 2`
Duplicate implementations of `calcStatusGeral`, `escapeHtml`, and numeric parsers (`parseNum` vs `parseNumeric`).
- **Risk:** Fixes applied to one instance (e.g., a bug in numeric parsing of commas) may not propagate to others, leading to inconsistent behavior across the app.

### 2.3 Silent Failures in Critical Detection
**Evidence:** `ResultadoDetalhe.tsx:370–378` (`avaliarNivelCritico`)
If a parameter lookup fails (e.g., mismatch between catalog name and parameter key), the function returns `"normal"` silently.
- **Risk:** Clinical safety gate fails open rather than failing closed or alerting the user.

---

## 3. Scalability Risks

### 3.1 Client-Side PDF Assembly
**Evidence:** `ResultadoDetalhe.tsx:768` (`buildLaudoHtml`)
PDFs are generated by concatenating large strings of HTML in the browser and then rendered/printed.
- **Risk:** Memory exhaustion or UI freezing when generating reports for patients with dozens of exams (e.g., intensive care patients).

### 3.2 Redundant Network Calls
**Evidence:** `ResultadoDetalhe.tsx:192-211`
The component re-loads layouts and parameters for every exam in the list on every "reload" trigger.
- **Risk:** Unnecessary DB load and latency as the number of exams per attendance grows.
